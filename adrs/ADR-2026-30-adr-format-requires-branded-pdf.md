# ADR-2026-30: All ADRs Require a Branded PDF

- **Status**: Superseded by [SOP-001](../sops/SOP-001-creating-adrs-and-sops.md)
- **Date**: 2026-03-11
- **Owner**: Daniel Shanklin / Director of AI & Technology
- **Related**: ADR-2026-05 (BLUF requirement), `reports/hubspot-dq/generate_report.py` (brand pattern reference)

## BLUF (Bottom Line Up Front)

Every ADR must ship with a branded PDF alongside the markdown. The PDF uses the Greenmark brand system (dark green header, logo, branded tables, accent boxes) generated via ReportLab. This ensures ADRs are presentation-ready for stakeholder meetings and maintain professional consistency across all project documentation.

## Context

ADRs serve two consumption modes at Greenmark:

1. **GitHub web UI** — developers and AI agents read the markdown directly in the repo
2. **Stakeholder presentations** — Michael, Alex, and external parties receive documents as PDFs in email, meetings, or shared drives

Markdown renders well in GitHub but looks unprofessional when printed or attached to an email. Meanwhile, Greenmark already has a proven branded PDF pattern (used in the HubSpot DQ report, work product reports, and now ADRs 04-05) that produces clean, consistent, presentation-grade documents.

The brand system is established:
- **Colors**: BRAND_GREEN `#193B2D`, ACCENT_GREEN `#2D6B4A`, BRAND_LIGHT `#E8F0EC`
- **Logo**: `infra/brand/greenmark-full-white.png` — white on dark green header
- **Fonts**: Helvetica family (built into ReportLab, no external dependencies)
- **Flowables**: BrandHeader, AccentBox, branded tables with alternating rows
- **Footer**: Company name, ADR title, date, page number

## Decision

**Every ADR must include a branded PDF generated by a co-located Python script.**

### Required artifacts per ADR:
| File | Purpose |
|------|---------|
| `ADR-YYYY-NN-slug.md` | Source of truth — markdown for GitHub and AI agents |
| `ADR-YYYY-NN-slug.pdf` | Branded PDF for stakeholders, meetings, and email |
| `generate_adr_NNN_pdf.py` | Generator script — reproducible, editable, version-controlled |

### PDF format requirements:
1. **BrandHeader** — dark green rounded rect with Greenmark logo, ADR number, and title
2. **BLUF section** — first content after metadata (per ADR-2026-05)
3. **Decision accent box** — red border for rejections, green border for acceptances
4. **Branded tables** — dark green header row, alternating white/light green body rows
5. **Footer** — company name, ADR title, date, page number on every page
6. **AccentBox callouts** — for key principles, warnings, or highlighted decisions

### Brand constants (copy from any existing generator):
```python
BRAND_GREEN = HexColor("#193B2D")
BRAND_LIGHT = HexColor("#E8F0EC")
ACCENT_GREEN = HexColor("#2D6B4A")
MUTED = HexColor("#6B7B73")
DARK = HexColor("#1A1A1A")
```

## Rationale

| Factor | Assessment |
|--------|-----------|
| **Stakeholder experience** | Michael and Alex receive polished documents, not raw markdown. Professional presentation builds confidence in the technical team. |
| **Consistency** | Every ADR looks the same. Brand recognition compounds across documents. |
| **Reproducibility** | Generator scripts are version-controlled. Re-run anytime to update. No manual formatting. |
| **Low cost** | ReportLab is the only dependency. Scripts are ~200 lines each. Pattern is well-established with 4 existing generators. |
| **AI-friendly** | Claude can generate and update these scripts autonomously. The pattern is documented in `memory/branded-pdf-reports.md`. |

## Options Considered

| Option | Description | Verdict |
|--------|-------------|---------|
| **A. Branded PDF required (chosen)** | ReportLab generator script per ADR | Professional, consistent, reproducible, low cost |
| B. Markdown only | Rely on GitHub rendering | Looks amateur when shared outside GitHub. Can't attach to emails. |
| C. Manual PDF export | Print-to-PDF from browser | Inconsistent formatting, no branding, not reproducible |
| D. LaTeX | Professional typesetting | Overkill. Heavy dependency. Harder for AI to generate. |
| E. Google Docs / Word | Manual document creation | Not version-controlled. Manual effort per ADR. Formatting drift. |

## Consequences

- **Positive**: Every ADR is presentation-ready from the moment it's committed. Stakeholders receive consistent, branded documents. Generator scripts serve as living documentation of the brand system.
- **Negative**: Each ADR requires an additional ~200-line Python script. ReportLab must be installed (`pip install reportlab`).
- **Mitigation**: The pattern is well-established — new generators are largely copy-paste from existing ones with content swapped. AI agents can generate them autonomously.
