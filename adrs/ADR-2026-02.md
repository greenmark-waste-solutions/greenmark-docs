# ADR-2026-02: Deterministic Checkpoint Architecture — Three-Tier Code Validation

- **Status**: Accepted
- **Date**: 2026-02-26
- **Owner**: Daniel Shanklin / Director of AI & Technology
- **Related**: [ADR-2026-01](ADR-2026-01.md), [Claude Code Hooks Guide](https://code.claude.com/docs/en/hooks-guide)

## Context

- AI agents (Claude Code) write and refactor code across multiple Greenmark repos (data-daemon, cerebro, cerebro-qa, rhea-diagrams). Agents are non-deterministic — they may forget to run tests, miss downstream references when refactoring, or introduce type errors that accumulate across edits.
- LSP plugins provide deterministic reference lookup but consume 2-4GB RAM continuously per language. On a laptop already running Claude Code, MCP servers, and browsers, this is prohibitive.
- The existing Python PostToolUse hook catches syntax errors and runs pytest, but has no type checking. JavaScript/TypeScript repos have zero hook coverage. No pre-commit configs exist in any Greenmark repo.
- The `bcpp` skill runs a build gate but doesn't explicitly gate on tests for Python repos.

## Decision

- Adopt a **three-tier deterministic checkpoint architecture** that fires type checking, tests, and build validation at specific moments — not when the agent remembers, but automatically at every relevant event.
- Use **event-driven hooks** (fire-check-exit) instead of **resident LSP processes** (always-on) to avoid the RAM cost.
- All three tiers are mandatory for Greenmark repos containing code.

### Tier 1: PostToolUse — Real-Time Type Checking

Fires after every `Edit` or `Write` tool call. Catches type errors as they're introduced.

| Language | Tool | What It Checks |
|----------|------|----------------|
| Python | `pyright` | Type errors, import resolution, undefined names |
| JavaScript/Node | `node --check` + `npm test` | Syntax validation + test suite |
| TypeScript | `tsc --noEmit` | Full type checking without emitting JS |

**Performance**: Pyright completes in ~100-200ms. `node --check` is instant. Zero memory cost between runs.

### Tier 2: Pre-Commit — Full Test Gate

Fires once per commit. Runs the full test suite. Cannot be bypassed (`--no-verify` is prohibited per CLAUDE.md).

| Repo | Gate |
|------|------|
| data-daemon | `pytest` (82 tests) |
| rhea-diagrams | `npm test` (180 tests) |
| cerebro | `npm run build` (type check + compile) |

### Tier 3: Skill-Level — bcpp Hard Gate

The `bcpp` (build-commit-push-PR) skill refuses to proceed unless tests pass. This is the highest-level enforcement — not advisory, mandatory.

## Rationale

- **Event-driven beats always-on.** Type checkers spin up, validate, exit. Zero RAM between edits. Same deterministic coverage as LSP without the 2-4GB resident cost.
- **Three tiers catch different error classes at optimal moments.** PostToolUse catches type errors in real-time (while the edit is in working memory). Pre-commit catches integration failures (before they enter the commit history). bcpp catches everything before code reaches the remote.
- **Mandatory beats optional.** The existing Python hook proves the pattern works. Expanding to all languages and all tiers closes the gaps.
- **Pre-commit hooks block 60% of bugs** before they reach the main branch (industry data).

## Options Considered

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **A. Three-tier hooks (chosen)** | PostToolUse + pre-commit + skill gate | Event-driven, zero idle RAM, deterministic | Requires per-repo configuration |
| B. LSP plugins | Always-on language servers | Real-time, full IDE features | 2-4GB RAM per language, performance issues |
| C. CI-only | Run checks only in GitHub Actions | Zero local overhead | Errors caught too late (after push) |
| D. Manual discipline | Trust the agent to run tests | No setup needed | Non-deterministic, agent forgets |

## Consequences

- **Positive**: Every edit is type-checked automatically. Every commit is test-gated. Every PR is build-verified. Agents cannot ship broken code without explicitly being told to bypass (which CLAUDE.md prohibits).
- **Negative**: PostToolUse hooks add ~100-500ms per edit. Pre-commit hooks add ~5-10s per commit. This is acceptable overhead for the reliability gain.
- **Neutral / Open**: TypeScript repos (cerebro) need `tsc` installed locally. Python repos need `pyright`. These are one-time installs.

1. PostToolUse hooks in `~/.claude/settings.json` for Python (pyright) and JavaScript (node --check + npm test)
2. Pre-commit configs in data-daemon and rhea-diagrams

## Review & Sunset

- **Review trigger**: After 30 days of usage, assess whether hook overhead is acceptable and whether false positives are disrupting workflow.
- **Conditions to revisit**: If Claude Code ships built-in type checking that makes PostToolUse hooks redundant. If LSP plugins solve their RAM problems.
