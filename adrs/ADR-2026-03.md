# ADR-2026-03: Supabase Migration Strategy — psycopg2 via Connection Pooler

- **Status**: Accepted
- **Date**: 2026-02-26
- **Owner**: Daniel Shanklin / Director of AI & Technology
- **Related**: greenmark-cockpit feedback system, cerebro Supabase integration

## Context

- Cerebro's feedback system is the first feature backed by Supabase (project: `greenmark-cerebro`, ref: `wwmcgtyngnziepeynccz`).
- We need a reliable way to run DDL migrations (CREATE TABLE, ALTER TABLE, storage bucket setup) against the remote Supabase database.
- The Supabase CLI (`v2.72.7`) does not have a `db execute` command — it only supports `db push` (for the migrations directory) and `db dump`.
- Direct connections to `db.*.supabase.co` resolve to IPv6 addresses that are unreachable from our development network (macOS, residential ISP). This is a hard blocker for `psql` and `psycopg2` using the direct endpoint.
- The Supabase connection pooler (Supavisor) is reachable via IPv4 and supports session mode on port 6543, which allows DDL and prepared statements.

## Decision

- **Run all Supabase migrations via a Python script using psycopg2 connected through the Supavisor session-mode pooler.** The script is at lib/supabase/migrate.py.
- Migration files live alongside the script at lib/supabase/migration-NNN-*.sql.
- The script reads the project ref from supabase/.temp/project-ref (set by supabase link) and accepts the database password via SUPABASE_DB_PASSWORD env var.
- We do NOT use the Supabase CLI migration system (supabase/migrations/ directory + db push) at this time.

## Rationale

- **IPv6 blocker is real.** Direct connections fail silently with "No route to host". This isn't a misconfiguration — it's a network topology reality. The pooler endpoint (aws-1-us-east-1.pooler.supabase.com:6543) resolves to IPv4 and works reliably.
- **psycopg2 is already installed.** No new dependencies needed. It handles transactions, error reporting, and DDL natively.
- **Numbered SQL files are simple and auditable.** Each migration is a plain .sql file. They can be reviewed in GitHub, run selectively by number, and are self-documenting.
- **The Supabase CLI migration system adds complexity we don't need yet.** It requires local Docker for db reset, manages a schema_migrations table, and is designed for teams with branching/staging workflows. We have one database and one developer.

## Options Considered

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **A. psycopg2 + pooler** (chosen) | Python script, numbered SQL files, session-mode pooler | Works now, simple, no Docker, auditable SQL files | No built-in tracking of which migrations have run |
| B. Supabase CLI db push | Official migration framework | Tracks applied migrations, branch support | Requires Docker locally, db execute doesn't exist in v2.72, direct connection fails |
| C. Supabase SQL Editor (browser) | Paste SQL into dashboard UI | Zero tooling setup | Manual, not reproducible, no audit trail, doesn't scale |
| D. psql direct connection | Use `psql` CLI with direct endpoint | Standard Postgres tooling | IPv6 routing failure, same network blocker as psycopg2 direct |

## Consequences

- **Positive**: Migrations work reliably from any network with IPv4. Script is version-controlled and shareable. Can run specific migrations by number. Zero Docker dependency.
- **Negative**: No automatic tracking of which migrations have been applied — the developer must know what's been run. Idempotent SQL (IF NOT EXISTS, ADD COLUMN IF NOT EXISTS) is required to make re-runs safe.
- **Neutral**: If we later adopt the Supabase CLI migration system (e.g., when adding staging environments), we can migrate the SQL files into supabase/migrations/ with timestamps. The SQL content is portable.

## Implementation & Ownership

- **Owner**: Daniel Shanklin
- **Location**: cerebro/lib/supabase/migrate.py + cerebro/lib/supabase/migration-\*.sql
- **Connection config**: Pooler host is aws-1-us-east-1.pooler.supabase.com:6543. This is hardcoded in the script. If the project moves regions, update POOLER_HOST.
- **Usage**: SUPABASE_DB_PASSWORD=xxx python3 lib/supabase/migrate.py \[NNN\]

## Lessons Learned (from initial setup)

1. **Always use the pooler endpoint** — never db.\{ref\}.supabase.co directly. IPv6 resolution + no route = silent failure.
2. **Pooler region matters** — this project is aws-1-us-east-1, not aws-0-us-east-1. Get the exact hostname from Project Settings > Database > Connection string.
3. **Username format for pooler** — must be postgres.\{ref\}, not just postgres. Supavisor uses this for tenant routing. Without the ref suffix: "Tenant or user not found".
4. **Port 6543 = session mode** (DDL-safe). Port 5432 on the pooler = transaction mode (no DDL, no prepared statements).
5. **Write idempotent migrations** — use IF NOT EXISTS, ADD COLUMN IF NOT EXISTS, etc. Since we don't track applied migrations, re-runs must be safe.
