# ADR-2026-01: Classify Claude Code as Tier-1 Engineering Infrastructure

- **Status**: Accepted
- **Date**: 2026-02-25
- **Owner**: Daniel Shanklin / Director of AI & Technology
- **Related**: Claude Code outage 2026-02-25; leadership bulletin on engineering tooling resilience

## Context

- Claude Code (Anthropic) has become the primary AI-assisted coding tool used across Greenmark engineering. It is embedded in daily development workflows for code generation, debugging, architecture, and documentation.
- On 2026-02-25, Claude Code experienced intermittent outages. During downtime, engineering velocity dropped materially — not just slower, but qualitatively different in how work could proceed.
- Several alternative AI coding tools exist (OpenAI Codex, Google Gemini CLI, and smaller niche offerings), but none currently match Claude Code in capability, workflow fit, or output quality for our use cases.
- This situation mirrors our dependency on other foundational platforms (AWS, Railway, GitHub) where no cogent 1:1 backup exists because the tool is central to operations, not peripheral.

## Decision

- Classify Claude Code as a **Tier-1, business-critical engineering system** — the same risk and dependency class as AWS, Railway, GitHub, and Bloomberg.
- Operate under a **single-primary with documented fallback** model rather than attempting to maintain parallel "dual primary" tooling.
- Maintain a written contingency plan ("break glass" sheet) that documents backup options, activation order, and expected capability tradeoffs if Claude Code becomes unavailable for an extended period.

## Rationale

- Claude Code delivers meaningfully better results than any available alternative for our engineering workflows. The gap is not marginal; it is significant enough that switching tools mid-sprint would cause a material productivity hit.
- Running dual-primary AI coding tools would split team expertise, increase onboarding cost, and dilute the workflow advantages of going deep on one platform.
- The correct analogy is Bloomberg on a trading desk: you rely on it as a centralized system, you respect its centrality, and you have a contingency plan — but you do not try to run a second Bloomberg in parallel.
- Concentration risk is real but is best managed through contingency planning, not through diluting the primary tool.

## Options Considered

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| A. Claude Code as Tier-1 (selected) | Single primary, documented fallback stack | Maximum velocity, deep team expertise, clear contingency | Concentration risk; outages hurt |
| B. Dual-primary (Claude + OpenAI) | Run two AI coding tools in parallel | Redundancy on paper | Splits expertise, doubles cost, neither tool gets fully adopted |
| C. Tool-agnostic / no default | Let engineers pick any tool | Maximum individual flexibility | No shared workflows, no institutional knowledge, inconsistent output |
| D. No AI coding tools | Revert to manual-only development | Zero vendor risk | Massive productivity regression; not competitive |

## Consequences

- **Positive**: Engineering team builds deep expertise on one platform. Workflows, prompts, conventions, and institutional knowledge compound over time. Leadership has clarity on what the engineering function depends on.
- **Negative**: When Claude Code is down, engineering throughput drops materially. We accept this concentration risk explicitly. Budget is concentrated in one vendor for this capability.
- **Neutral / Open**: We are not deciding on a specific secondary tool right now — only that we will document ranked fallback options. We are also not deciding on abstraction layers or wrapper tooling at this time.

## Implementation & Ownership

- **Owner**: Daniel Shanklin (Director of AI & Technology)
- **Action items**:
  1. Create and maintain a "break glass" backup sheet listing fallback tools in priority order: (1) OpenAI Codex, (2) Google Gemini CLI, (3) other niche tools. Include expected capability gaps for each.
  2. Communicate this classification to leadership so Claude Code outages are understood as infrastructure incidents, not minor tool inconveniences.
  3. Monitor the AI coding tool landscape quarterly and update the backup sheet as alternatives mature.
- **Success metric**: When Claude Code has an outage, leadership understands the impact without needing ad hoc explanation, and the team can activate fallback workflows within 1 hour using the documented plan.

## Review & Sunset

- **Review trigger**: Quarterly, or immediately upon any of the following:
  - Claude Code outage exceeding 48 hours
  - Anthropic pricing increase exceeding 50%
  - A competing tool demonstrably closing the capability gap in a head-to-head evaluation
  - Material change in Anthropic's business stability or product direction
- **Conditions to revisit**: If any alternative tool reaches approximate parity with Claude Code for our workflows, re-evaluate whether the single-primary model still holds or whether dual-primary becomes viable.
